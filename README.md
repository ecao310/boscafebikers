# Boston Café Bikers

Exploring the city one café at a time.

A casual, no-drop Boston cycling group that rides to cafés. This repo is the
group's website: a single static page whose ride schedule is auto-synced every
6 hours from the organizer's Partiful calendar feed.

- Site: `site/index.html` (one file — inline CSS/JS, no build step, no frameworks)
- Ride data: `site/events.json`, generated by `scripts/fetch_rides.py`
- Sync: `.github/workflows/sync.yml` (cron every 6h + manual run)
- Deploy: `.github/workflows/pages.yml` → <https://ecao310.github.io/boscafebikers/>
- Instagram: [@bostoncafebikers](https://instagram.com/bostoncafebikers)
- Partiful: <https://partiful.com/u/Hs47uq5mucZyXLBJZCda>

## How it works

```
Partiful ICS feed  →  scripts/fetch_rides.py  →  site/events.json  →  site/index.html
   (repo secret)         (icalendar parse)        (committed by CI)      (fetch + render)
```

`fetch_rides.py` keeps only events starting at or after "now" in
`America/New_York`, drops `STATUS:CANCELLED` ones, pulls the `RSVP: <url>` line
out of each description, sorts by start time, and precomputes the display
strings so the page never re-formats dates in the visitor's timezone.

If `events.json` is missing, empty, or unreadable, the page falls back to a
link to the Partiful profile instead of showing an empty list.

## Getting the ICS URL from Partiful

1. Open Partiful and go to **Settings → Calendar Sync → Apple Calendar**.
2. Copy the subscription URL. It starts with `webcal://`.
3. Swap the scheme: `webcal://` → `https://`. (The script does this rewrite
   itself too, so either form works.)

**Never commit this URL.** It is effectively a private feed of the group's
events. It belongs only in the `PARTIFUL_ICS_URL` environment variable / GitHub
secret. The fetch script is written to never print it, even inside error
messages.

## Setting the repo secret

GitHub → your fork/repo → **Settings → Secrets and variables → Actions →
New repository secret**:

- Name: `PARTIFUL_ICS_URL`
- Value: the `https://` URL from above

The workflow passes it to the fetch step as an env var (never as a CLI
argument, since arguments show up in workflow logs).

> **This secret is not set yet on `ecao310/boscafebikers`.** Until someone with
> the real feed URL adds it, every scheduled sync fails at the fetch step and the
> committed `site/events.json` stays at its current contents. It is the one
> remaining manual setup step.

To sync immediately rather than waiting for the cron: **Actions → Sync rides →
Run workflow**.

## Deploying on GitHub Pages

**Live: <https://ecao310.github.io/boscafebikers/>**

The site is deployed by `.github/workflows/pages.yml` using the **GitHub
Actions** Pages source — `actions/upload-pages-artifact` with `path: site`, then
`actions/deploy-pages`. The "Deploy from a branch" source can't be used here: it
only offers `/` or `/docs` as the publish folder, and the site lives in `site/`.

One-time setup on a fresh fork:

1. GitHub → **Settings → Pages → Source: GitHub Actions**. Headless equivalent:
   ```bash
   gh api -X PUT repos/:owner/:repo/pages -f build_type=workflow
   ```
2. Push to the default branch (`master` here). That's it — `pages.yml` runs on
   `push` to the default branch and on `workflow_dispatch`.

Redeploy by hand:

```bash
gh workflow run pages.yml --ref master
gh run watch <run-id>
```

**Staying fresh.** Commits made by a workflow's `GITHUB_TOKEN` do not fire
`push` triggers, so the sync bot's commit can't redeploy the site that way.
Instead `pages.yml` is `workflow_call`-able, and `sync.yml` has a `deploy` job
that calls it — only when `promote_events.py` actually changed the rides. If the
feed is unchanged, no commit and no deploy.

The site is served from `https://<user>.github.io/<repo>/`, a project subpath.
Every URL in `index.html` is relative or absolute-with-host — nothing starts
with `/` — so it works there without config. Keep it that way; a root-relative
path would 404.

**Custom domain (future).** Add a `site/CNAME` file containing the bare domain
(e.g. `bostoncafebikers.com`) so it ships with the artifact and survives every
redeploy, then set the same domain under **Settings → Pages → Custom domain**
and point DNS at GitHub Pages.

## Local development

```bash
python3 -m venv .venv
.venv/bin/pip install -r requirements.txt
```

Run the fetch script against the offline fixture (never the live feed during
development or tests):

```bash
.venv/bin/python scripts/fetch_rides.py --ics-file tests/fixtures/sample.ics
```

Preview the site:

```bash
.venv/bin/python -m http.server -d site 8000   # → http://localhost:8000
```

(Open it over HTTP, not `file://` — the `fetch("events.json")` call needs a
real origin.)

Run the tests:

```bash
.venv/bin/python -m pytest tests/ -q
```

The test suite is fully offline: it parses `tests/fixtures/sample.ics` with a
pinned clock, so it never touches the network and never depends on the system
timezone.
